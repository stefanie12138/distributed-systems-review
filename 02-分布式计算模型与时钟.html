第3-4讲：分布式计算模型与时钟

一、知识点讲解

1.1 消息传递模型

1.1.1 含义

消息传递模型（Message Passing Model）是分布式系统设计中最基础的抽象模型。在消息传递模型中：

  • 分布式系统由多个独立的计算节点（进程/主机）组成
  • 每个节点拥有自己独立的地址空间（独立内存）
  • 节点之间不共享内存，只能通过发送和接收消息来进行通信和协作
  • 消息通过通信网络（如以太网、InfiniBand、无线网络等）在节点之间传输
  • 每个节点只知道自己本地的状态，没有全局视图——只能通过与其它节点的消息交互来推断整个系统的状态

1.1.2 与多线程并行计算模型的区别

多线程并行计算系统（共享内存模型）与分布式消息传递模型的核心区别：

维度 消息传递模型（分布式） 共享内存模型（多线程）
内存模型 每个节点拥有独立的地址空间，无共享内存 所有线程共享同一地址空间
通信方式 通过网络发送/接收消息 通过共享变量读写进行通信
同步机制 消息收发本身隐含同步（或分布式锁等） 使用互斥锁、信号量、条件变量等
节点失效 部分节点失效是常态，必须考虑容错 通常假设不发生线程"崩溃"，或直接导致进程崩溃
延迟 消息传输延迟不可忽略，且剧烈变化 共享内存访问延迟极低且相对稳定
扩展方式 水平扩展（增加节点数），可跨越物理机器 受限于单机CPU核心数和内存带宽

关键结论： 在消息传递模型中，设计者必须时刻考虑网络不可靠性、部分节点失效、时钟不同步等问题，而这些在单机多线程模型中通常不是首要问题。

1.2.4 产品级分布式系统的挑战

设计产品级（可用于实际生产环境）的分布式系统需要关注的特性：

  • 功能正确性：系统逻辑是否正确实现
  • 效率：时间、空间、通信效率
  • 可扩展性：
  • 垂直可扩展性：升级单节点硬件（加内存、CPU等）
  • 水平可扩展性：增加节点数量，支持节点热插拔
  • 容错性：包含可用性（Availability）和可恢复性（Recoverability）
  • 透明性：对用户隐藏分布式系统的复杂性（如访问透明、位置透明、迁移透明、复制透明等）
  • 开放性：系统能否方便地与其他系统互操作
  • 安全性：防止未授权访问和攻击
  • 可维护性：系统是否易于调试、升级和维护

设计分布式系统面临的四大不可靠因素：

  1. 不可靠的计算节点：部分节点失效是常态（而非异常）
  2. 不可靠的通信网络：丢包、乱序、传输延迟剧烈变化、数据包内容被篡改、网络断裂
  3. 不可靠的时钟：不同节点的物理时钟不同步
  4. 没有全局观：只能通过与其他节点的交互信息对系统状态进行评估

1.2.5 两将军问题与拜占庭将军问题

两将军问题（The Two Generals Problem）：

  • 两支军队分驻两座山头，需要通过信使通信来协调同时进攻时间
  • 信使需要穿过敌方山谷，可能被俘（消息丢失）
  • 该问题从理论上证明：在基于不可靠通信网络的分布式系统中，两节点无法100%就某事达成共识
  • 工程启示：分布式事务、分布式共识协议必须放宽假设（如允许超时重试、多数派投票等）

拜占庭将军问题（The Byzantine Generals Problem）：

  • 多支军队围城，将军之间通过信使通信，但部分将军可能是叛徒（恶意节点）
  • 叛徒会发送矛盾、欺骗性的消息来扰乱共识
  • 该问题说明：在存在恶意节点的不可靠网络中达成共识更加困难
  • 工程应用：开放分布式系统中的安全设计；区块链（如比特币）中的共识机制

1.3 网络模型（链路模型）

以下主要考虑点对点通信拓扑，从弱到强分为四种链路模型：

1.3.1 任意链路（Arbitrary Link）

  • 最弱的模型：丢包、重复包、内容篡改、伪造、乱序、传输延迟抖动
  • 存在主动攻击者（恶意篡改、伪造消息）
  • 对应现实中的开放公网环境

如何基于任意链路实现一般损失链路： - 利用纠错码检测/纠正传输错误 - 利用哈希函数（HASH）检测数据完整性 - 利用消息认证码（MAC）检测消息是否被篡改 - 利用加密算法防止内容被窃取和伪造 - 利用 SSL/TLS/HTTPS 协议提供传输层安全保障

1.3.2 一般损失链路（Fair-loss Link）

  • 只存在丢包、重复包、乱序和传输延迟抖动
  • 无主动攻击者（无内容篡改、无伪造）
  • 链路可能发生断裂（Partition），但断裂持续时间是有限的

如何基于一般损失链路实现可靠链路： - 发送者持续重传数据包，直到收到ACK确认 - 利用数据包序列号去除重复数据包 - 接收者用序列号来检测重复和乱序

1.3.3 可靠链路（Reliable Link）

  • 发送者发出的每个消息都可以被正确接收
  • 但是：（1）存在传输延迟抖动；（2）接收到消息的顺序可能与发送顺序不一致
  • 相当于保证了"最终一定到达"，但不保证顺序

1.3.4 可靠FIFO链路（Reliable FIFO Link）

  • 最强模型：发出的每个消息都可以被正确接收，且不会乱序
  • 消息按发送顺序被接收

如何基于可靠链路实现可靠FIFO链路： - 利用数据包序列号，对收到的数据包进行排序 - 按序列号由小到大依次向应用层投递

重要注释： 严格来讲，简单地依赖TCP协议不能实现真正的"可靠FIFO链路"——TCP连接断裂后，需要在此基础上实现TCP链接失效后的持续重新建立连接和重传机制。

1.4 节点故障模式

1.4.1 失效停止模式（Fail-Stop Failure / Crash Failure）

  • 节点一旦发生故障就停止工作
  • 故障期间不发送任何消息（安静地停止）
  • 变体：失效恢复模式（Fail-Recovery）——节点崩溃后可以重新启动并再次上线。崩溃前内存中的状态全部丢失，但持久化到非易失性存储器（如硬盘、SSD）中的状态还可以恢复
  • 这是最常用的故障假设模型，许多分布式协议基于此模型设计

1.4.2 拜占庭模式（Byzantine Failure / Fail-Arbitrary）

  • 节点发生故障后可能做出任意动作（"胡乱作为"）
  • 例如：发送非协议规定的错误消息、故意发送矛盾消息来扰乱系统
  • 网络安全系统设计多采用拜占庭模型——此时认为节点完全被黑客控制
  • 一些安全系数要求极高的分布式系统设计中也采用拜占庭模型（如航天系统、金融系统）

1.4.3 两种模式的对比

维度 失效停止模式 拜占庭模式
故障行为 停止工作，不发消息 可能发送任意错误/恶意消息
故障原因 硬件故障、进程崩溃、断电 黑客攻击、软件Bug、恶意节点
协议复杂度 相对简单 极其复杂
容错要求 n > f（或 n > 2f，取决于协议） n > 3f
典型应用 大多数分布式系统 区块链、安全关键系统

1.5 时间（同步）模式

1.5.1 同步模式（Synchronous）

  • 消息传输延迟不超过已知的上限（存在已知的最大网络延迟）
  • 节点以已知速度执行算法（每个计算步骤的执行时间有已知上限）
  • 优点：协议设计简单，可以使用超时机制检测故障
  • 缺点：现实中很难保证严格同步

1.5.2 异步模式（Asynchronous）

  • 消息传输延迟没有上限（可能无限长）
  • 节点在运行期间可能会暂停任意长的时间（如被操作系统调度挂起、GC暂停等）
  • 没有任何时间假设可用
  • FLP不可能性结论：在纯异步模型中，即使只有一个节点可能失效，也不存在确定性的共识算法

1.5.3 部分同步模式（Partially Synchronous）

  • 系统大部分时间工作在同步模式
  • 在有限时间段内（该时间段长度不可预知）工作在异步模式
  • 但是系统最终会回到同步模式
  • 工程应用：设计分布式协议时可以在某些环节设定超时值，但协议的正确性（safety）和活性（liveness）不完全依赖于该超时值
  • 这是最接近现实世界的模型，也是Paxos、Raft等协议的基础假设

1.6 物理时钟

1.6.1 物理时钟基础

每个计算机都带有基于晶振的计时器（频率发生器），根据该计时器生成的时钟称为物理时钟。部分计算机还带有电池，使得在关机期间计时器仍然在工作。

更精确的物理时钟源： - 铯原子钟（Cesium Atomic Clock） - GPS时钟（GPS卫星携带原子钟）

物理时钟分类：

类型 单调钟（Monotonic Clock） 墙钟（Wall-Clock Time）
行为 按一定频率持续自增，永不回退 对应日常日历时间/天文事件
值的含义 绝对值无意义，相对值表示延时/时间差 绝对值有意义（年月日时分秒）
精度 较高（纳秒级），单调递增 可能发生倒流（NTP校正、闰秒）
Java API System.nanoTime() System.currentTimeMillis()
用途 测量时间间隔、性能测量、超时 日志时间戳、用户可见时间
标准 无统一标准 UTC（协调世界时）

System.currentTimeMillis() 返回自1970年1月1日午夜0点以来的毫秒数（Unix时间戳）。

1.6.2 物理时钟的同步问题

  • 每个节点有自己独立的物理时钟
  • 由于硬件问题（晶振制造公差），不同节点的物理时钟精度不一样
  • 节点的物理时钟还容易受到温度、使用年限、震动、辐射等环境因素影响
  • 即便开始做了时钟同步，随着时间的推移，不同节点物理时钟之间的差异逐渐增大，最后超出误差范围（失同步）
  • 与NTP服务进行时间同步时带来的误差约为几十到一百毫秒

参考数据： - 每30秒与NTP服务器重新同步一次的时钟漂移为6毫秒 - 每天与NTP服务器重新同步一次的时钟漂移为17秒

1.6.3 实现物理时钟同步的方法

方法一：直接与世界标准时间（UTC）同步 - 世界上建有40多个地面短波站，每秒向全世界广播UTC时间 - GPS接收器也可以接收到UTC时间 - 缺点：（1）成本高；（2）大部分计算机在室内，接收不到信号

方法二：通过网络与时间服务器同步 - 时间服务器配备GPS接收器或其他高精度时钟源 - 节点定期通过网络与时间服务器同步 - 需要考虑网络传输延时

1.7 Cristian时间同步算法与NTP

1.7.1 Cristian时间同步算法

该算法是NTP（Network Time Protocol）的核心基础。设客户端C向服务器S请求时间同步：

步骤：

  1. 客户端在本地时间 $$T_1$$ 发送请求包给服务器
  2. 服务器在本地时间 $$T_2$$ 收到请求（服务器记录 $$T_2$$）
  3. 服务器在本地时间 $$T_3$$ 发送响应包（响应包携带 $$T_2$$ 和 $$T_3$$）
  4. 客户端在本地时间 $$T_4$$ 收到响应包

推导过程：

网络往返传输延迟（Round-Trip Time, RTT）：

$$\delta = \delta_{req} + \delta_{resp} = (T_4 - T_1) - (T_3 - T_2)$$

其中： - $$\delta_{req} = T_2 - T_1$$（请求传输延迟，未知） - $$\delta_{resp} = T_4 - T_3$$（响应传输延迟，未知） - 客户端知道总延迟 $$\delta$$，但不知道 $$\delta_{req}$$ 和 $$\delta_{resp}$$ 各占多少

关键假设： $$\delta_{req} \approx \delta_{resp}$$（请求和响应的网络延迟大致相等）

在此假设下，$$\delta_{resp} \approx \frac{\delta}{2}$$

因此，客户端在收到服务器响应时，服务器时间约为：

$$T_{server\_now} \approx T_3 + \frac{\delta}{2}$$

客户端将本地时钟设置为：

$$T_{client} \leftarrow T_3 + \frac{\delta}{2}$$

时钟偏移（Clock Skew）： 由于 $$\delta_{req}$$ 和 $$\delta_{resp}$$ 不一定相等，实际同步误差最大为 $$\pm \frac{\delta}{2}$$。即：

$$|T_{client\_corrected} - T_{server\_actual}| \leq \frac{\delta}{2}$$

1.7.2 NTP本地时钟调整策略

NTP采用三种策略调整本地时钟：

  1. 微调本地时钟的频率（Slewing）： 稍微加快或减慢本地时钟的频率，使本地时钟逐渐追上正确时间。这是最常用的方法，不会造成时间跳跃。
  2. 直接设置本地时钟的值（Stepping）： 当本地时钟与正确时间相差太大（如超过128ms）时，直接跳变到正确时间。缺点是会导致时间不连续（倒流或跳变）。
  3. 直接向用户报错： 当时钟差距过大且无法自动修复时，通知系统管理员。

1.7.3 常用的时钟同步协议

协议 全称 精度 应用场景

1.8 Lamport逻辑时钟（标量时钟）

1.8.1 为什么需要逻辑时钟

  • 物理时钟很难做到精确同步，因此在分布式系统中无法用物理时钟对事件的发生进行排序
  • 分布式系统中很多问题的关键在于：不同节点对多个事件的发生顺序达成一致即可，且达成一致的顺序尽量不破坏因果关系
  • Leslie Lamport 在 1978 年的论文《Time, Clocks, and the Ordering of Events in a Distributed System》首次提出了逻辑时钟的思想，并因此获得2013年图灵奖

1.8.2 先于关系（Happen-Before Relation）

定义"先于"关系，记作 $$\rightarrow$$，是一个偏序关系：

  1. 同一进程内： 如果事件 $$a$$ 和 $$b$$ 是同一个节点（单进程/线程）中的两个事件，且 $$a$$ 在 $$b$$ 之前发生，那么 $$a \rightarrow b$$
  2. 消息传递： 如果 $$a$$ 是"节点 $$P_1$$ 发送消息 $$m$$ 到 $$P_2$$"，$$b$$ 是"$$P_2$$ 收到 $$P_1$$ 发送的消息 $$m$$"，那么 $$a \rightarrow b$$
  3. 传递性： 如果 $$a \rightarrow b$$ 且 $$b \rightarrow c$$，那么 $$a \rightarrow c$$
  4. 并发事件： 如果 $$a \nrightarrow b$$ 且 $$b \nrightarrow a$$，则 $$a$$ 和 $$b$$ 是并发事件（记作 $$a \parallel b$$）

关键性质： - $$\rightarrow$$ 关系符合离散数学中的偏序定义 - $$\rightarrow$$ 关系隐含了因果关系（如果 $$a \rightarrow b$$，则 $$a$$ 可能因果影响了 $$b$$）

1.8.3 Lamport逻辑时钟算法

基本思想： - 每个节点维护一个不断增加的整数作为本地逻辑时钟 - 节点内事件的发生会触发逻辑时钟的增长

算法规则（设 $$C_i$$ 为节点 $$i$$ 的本地逻辑时钟）：

  1. 内部事件/发送事件： 在节点 $$i$$ 中，每发生一个新事件（包括消息产生、消息发送），本地逻辑时钟自增： $$C_i \leftarrow C_i + 1$$

  2. 发送消息： 节点 $$i$$ 发出的每个消息都绑定当前的 $$C_i$$ 作为时间戳

  3. 接收消息： 当节点 $$j$$ 接收到节点 $$i$$ 发送的消息 $$m$$（时间戳为 $$C_i$$），节点 $$j$$ 将自己的逻辑时钟调整为： $$C_j \leftarrow 1 + \max\{C_j, C_i\}$$

解释： 接收方将自己的时钟设为"自身时钟和消息时间戳的较大值加1"。这保证了接收事件的时间戳一定大于发送事件的时间戳。

1.8.4 Lamport逻辑时钟的性质

设 $$C(a)$$ 表示事件 $$a$$ 对应的逻辑时间戳：

  1. 保持因果关系： 如果 $$a \rightarrow b$$，则一定有 $$C(a) < C(b)$$

  2. 即：逻辑时钟从不违反因果关系

  3. 反过来不成立： $$C(a) < C(b)$$ 时，不一定满足 $$a \rightarrow b$$

  4. $$a$$ 和 $$b$$ 可能是并发事件（没有因果关系），但巧合使得 $$C(a) < C(b)$$

  5. 这是标量时钟的根本局限

  6. 可能相等： 有可能 $$a \neq b$$ 但 $$C(a) = C(b)$$（两个并发事件可能获得相同的时间戳）

1.8.5 逻辑时钟中间件的位置

实现逻辑时钟功能的软件模块一般放在中间件中： - 位于应用层和OS网络协议栈层之间 - 对上为应用程序提供API接口 - 对下调用网络协议栈层的服务接口（Socket接口） - 也可以与应用层集成到一起

1.9 全序逻辑时钟

1.9.1 动机

原始Lamport逻辑时钟存在一个问题：当 $$C(a) = C(b)$$ 时，无法区分两个事件的先后顺序。全序逻辑时钟将时间戳从标量扩展为二元组，实现所有事件的全序排列。

1.9.2 全序逻辑时钟算法

时间戳格式： 每个消息/事件携带二元组 $$\langle C_i, i \rangle$$，其中 $$C_i$$ 是标量逻辑时钟值，$$i$$ 是节点ID号。

排序规则： 对于两个事件 $$a$$ 和 $$b$$，其时间戳分别为 $$L(a) = \langle C(a), ID(a) \rangle$$ 和 $$L(b) = \langle C(b), ID(b) \rangle$$：

$$L(a) < L(b) \iff C(a) < C(b) \;\text{OR}\; \big(C(a) = C(b) \;\text{AND}\; ID(a) < ID(b)\big)$$

算法描述： - 每个节点维护标量时钟 $$C$$ - 节点内每发生新事件，$$C \leftarrow C + 1$$ - 发送消息时携带 $$\langle C, ID \rangle$$ 时间戳 - 接收到节点 $$i$$ 的消息（时间戳 $$\langle C_i, i \rangle$$）后：$$C_j \leftarrow 1 + \max\{C_j, C_i\}$$

排序的性质： - 保持了潜在因果关系：$$a \rightarrow b$$ 则一定有 $$L(a) < L(b)$$ - 但反过来不成立（与标量时钟相同）

缺点： 如果 $$L(a) < L(b)$$，无法区分以下两种情况： - $$a \rightarrow b$$（有因果关系） - $$a \parallel b$$（逻辑并发，无因果关系）

这个缺点引出了向量时钟。

1.10 向量逻辑时钟

1.10.1 基本思想

设分布式系统包含 $$n$$ 个节点 $$N_1, N_2, \ldots, N_n$$。

节点 $$N_i$$ 的向量时钟 $$\mathbf{T}_i$$ 形式为： $$\mathbf{T}_i = \langle t_1, t_2, \ldots, t_n \rangle$$

其中： - $$\mathbf{T}_i[i] = t_i$$：节点 $$N_i$$ 自己的标量逻辑时钟 - $$\mathbf{T}_i[j]\;(j \neq i)$$：节点 $$N_i$$ 所知道的关于节点 $$N_j$$ 的最新（最大）标量时钟值

直观理解： 向量时钟相当于让每个节点维护了一个关于"我认为每个节点现在进展到哪一步了"的全局视图。

1.10.2 向量时钟算法

  1. 初始化： 所有节点的向量时钟初始化为 $$\langle 0, 0, \ldots, 0 \rangle$$

  2. 内部事件： 节点 $$N_i$$ 内每发生一个新事件： $$\mathbf{T}_i[i] \leftarrow \mathbf{T}_i[i] + 1$$

  3. 发送消息： 节点 $$N_i$$ 发送消息时，将当前的向量时钟 $$\mathbf{T}_i$$ 作为时间戳附加到消息上

  4. 接收消息： 节点 $$N_i$$ 收到来自节点 $$j$$ 的消息 $$\langle m, \mathbf{T}_m \rangle$$ 后：

  5. 融合： 对于所有 $$k = 1, 2, \ldots, n$$： $$\mathbf{T}_i[k] \leftarrow \max(\mathbf{T}_i[k], \mathbf{T}_m[k])$$
  6. 自增： $$\mathbf{T}_i[i] \leftarrow \mathbf{T}_i[i] + 1$$（接收也是一个事件）

1.10.3 向量时钟的比较与排序

对于任意的两个向量时钟 $$\mathbf{T}$$ 和 $$\mathbf{T}'$$，定义如下偏序关系：

  • 相等： $$\mathbf{T} = \mathbf{T}' \iff \forall j: \mathbf{T}[j] = \mathbf{T}'[j]$$
  • 小于等于： $$\mathbf{T} \leq \mathbf{T}' \iff \forall j: \mathbf{T}[j] \leq \mathbf{T}'[j]$$
  • 严格小于： $$\mathbf{T} < \mathbf{T}' \iff \mathbf{T} \leq \mathbf{T}' \text{ 且 } \mathbf{T} \neq \mathbf{T}'$$
  • 并发（不可比较）： $$\mathbf{T} \parallel \mathbf{T}' \iff \mathbf{T} \nleq \mathbf{T}' \text{ 且 } \mathbf{T}' \nleq \mathbf{T}$$

1.10.4 向量时钟的关键性质

设 $$V(a)$$ 表示事件 $$a$$ 的向量时间戳：

  1. $$a \rightarrow b \iff V(a) < V(b)$$
  2. 这是向量时钟相对于标量时钟的核心优势：因果关系和向量时钟大小的关系是充要的

  3. 不再是单向蕴含，而是双向等价！

  4. $$a \parallel b \iff V(a) \parallel V(b)$$（即 $$V(a)$$ 和 $$V(b)$$ 不可比较）

  5. 可以通过向量时钟精确判断两个事件是否并发

标量时钟 vs 向量时钟对比：

性质 标量时钟（Lamport） 全序标量时钟 向量时钟
时间戳类型 整数 二元组〈C, ID〉 n维向量
排序类型 偏序 全序 偏序
保持因果关系 ✅（单向） ✅（单向） ✅（充要）
检测并发 ❌ ❌ ✅
存储开销 O(1) O(1) O(n)
通信开销 O(1) O(1) O(n)

1.11 广播协议

广播在分布式系统中是指一个节点发送消息，所有节点（或特定组内所有节点）都能收到该消息。

1.11.1 可靠广播协议（Reliable Broadcast）

性质：

  1. 有效性（Validity）： 如果一个正确节点广播了消息 $$m$$，则其它所有正确节点最终都能正确接收到 $$m$$
  2. 一致性/统一性（Uniformity/Agreement）： 如果消息 $$m$$ 被正确投递（Delivered）给某个正确节点 $$A$$，则 $$m$$ 一定会被正确投递给所有其它非失效节点
  3. 完整性（Integrity）： 如果一个正确节点接收到了广播消息 $$m$$，则 $$m$$ 一定是某个节点发送的（不能"无中生有"）

1.11.2 全序广播协议（Total Order Reliable Broadcast）

在可靠广播的基础上，再满足： - 全序性： 多个广播消息以完全相同的顺序投递给所有节点（所有节点以完全相同的顺序接收到多个广播消息）

1.11.3 先进先出广播协议（FIFO Reliable Broadcast）

在可靠广播的基础上，再满足： - FIFO序： 如果同一个节点先广播了消息 $$m_1$$，后广播了 $$m_2$$，则在其它所有非失效节点上，$$m_1$$ 先投递，$$m_2$$ 后投递

1.11.4 因果序广播协议（Causal Reliable Broadcast）

在可靠广播的基础上，再满足： - 因果序： 如果两个消息 $$m_1$$ 和 $$m_2$$ 有因果关系（$$m_1 \rightarrow m_2$$），则在任意节点上 $$m_1$$ 一定会先于 $$m_2$$ 投递给应用层

1.11.5 Gossip广播（传染病广播协议）

主要思想： 当一个节点首次接收到一个消息 $$m$$ 之后，它随机选取 $$k$$ 个节点（$$k$$ 一般为常数，如3~4），然后将 $$m$$ 转发给这 $$k$$ 个节点。

特点： - 基于六度分隔理论（Six Degrees of Separation） - 理论上无法实现100%的可靠广播（消息可能因为随机选择而遗漏某些节点） - 通过调整参数 $$k$$，可以在可靠性和通信开销之间进行权衡 - 去中心化、简单、容错性好、可扩展性好

典型应用场景： - 分布式系统中节点之间的相互发现（Membership） - 关键状态信息的传播（如节点加入/离开信息） - Cassandra、Dynamo等分布式数据库中的状态传播

1.11.6 不同类型广播协议之间的关系

广播协议从弱到强的关系：

  • Best-Effort Broadcast（尽力而为广播）：最弱，发送者失效则广播失败
  • Eager Reliable Broadcast：每个收到消息的节点都向所有其他节点转发
  • Gossip Broadcast：每个节点随机选择k个节点转发
  • Reliable Broadcast：保证所有节点最终收到
  • FIFO Reliable Broadcast：Reliable Broadcast + 同一发送者的消息保序
  • Causal Reliable Broadcast：Reliable Broadcast + 因果关系保序
  • FIFO Total Order Reliable Broadcast：Reliable Broadcast + FIFO + 全序
  • Total Order Reliable Broadcast：最强的广播协议，所有消息全局统一顺序

1.12 全序广播的实现与分布式互斥锁

1.12.1 全序广播协议实现（可靠FIFO链路 + 无节点失效）

假设模型： 点对点拓扑；可靠FIFO链路；节点不会失效

每个节点执行如下协议：

  1. 广播消息： 需要广播时，将消息利用点对点链路发送给所有节点（包括自己）
  2. 收到消息：
  3. （a）将消息按逻辑时间戳先后顺序放入缓存队列
  4. （b）向所有节点发送关于该消息的 ACK 消息（包括自己）
  5. 收到ACK消息：
  6. （a）将队列中对应消息 $$m$$ 的 ACK 计数加 1
  7. （b）观察队列头部的消息，如果关于该消息已经收到了所有节点的ACK，则将其从队列中取出，完成投递（给应用层）

定理： 如果缓存队列中处于头部的消息收到了所有节点的ACK，则该消息也位于所有节点内部队列的头部。（可用反证法证明）

优化方法： 节点 $$N_2$$ 收到 $$N_1$$ 的广播消息 $$m_1$$ 后，如果 $$N_2$$ 也有自己的消息 $$m_2$$ 要广播，则可以直接广播 $$m_2$$，不用广播关于 $$m_1$$ 的 ACK 消息。

1.12.2 全序广播协议实现（可靠链路 + 无节点失效）

假设模型退化为"可靠链路"（不保证FIFO）时，协议需做如下调整：

  • 步骤(2)(b)改为：观察缓存队列头部的消息，如果关于该消息尚未广播过ACK，则向所有节点发送关于该消息的ACK消息

这样避免了对同一消息重复发送ACK，同时通过ACK的完整性保证全序。

1.12.3 用全序广播实现分布式互斥锁

问题： 如何在分布式系统中保证同一时刻只有一个节点可以访问某共享资源？

假设模型： 点对点拓扑；可靠FIFO链路；节点不会失效

协议流程：

  1. 想要访问共享资源的节点利用可靠广播协议广播一个 LOCK 消息
  2. 任意节点收到 LOCK 消息后：
  3. (i) 将 LOCK 消息放入本地队列，按逻辑时间戳排序
  4. (ii) 利用可靠广播协议广播一个针对该 LOCK 的 ACK 消息
  5. 任意节点收到 ACK 消息后：
  6. (i) 在队列中找到对应的 LOCK 消息，将 ACK 计数加 1
  7. 获得锁的条件（由申请锁的节点检查）：
  8. (i) 自己发出的 LOCK 消息是否位于本地队列的头部
  9. (ii) 自己发出的 LOCK 消息是否已经收到了所有节点的 ACK
  10. 两个条件都成立 → 成功获得互斥锁，开始访问共享资源
  11. 获得锁的节点访问完共享资源后，广播一个 RELEASE 消息
  12. 任意节点收到 RELEASE 消息后：删除队列头部对应的 LOCK 消息
  13. 等待获得锁的节点收到 RELEASE 后，重新检查条件 (4)

1.13 分布式共识协议（Paxos与Raft）

1.13.1 共识问题的背景

在分布式系统中，多个节点需要就某个值（如谁是主节点、下一个要执行的操作是什么）达成一致。由于网络不可靠、节点可能失效，达成共识是分布式系统的核心难题。

1.13.2 Paxos协议

Paxos 是由 Leslie Lamport 提出的分布式共识算法（1998年发表，是逻辑时钟同一作者）。

基本思想（三阶段）： 1. Prepare（准备）阶段： 提议者（Proposer）向多数派节点发送提案号 2. Promise（承诺）阶段： 接受者（Acceptor）承诺不再接受更小编号的提案，并返回已接受的最高编号提案 3. Accept（接受）阶段： 提议者收集到多数派的承诺后，发送接受请求；多数派接受后，提案被选定（Chosen）

Paxos 中的角色： - Proposer（提议者）： 发起提案 - Acceptor（接受者）： 对提案进行投票 - Learner（学习者）： 学习已确定的提案值

1.13.3 Raft协议

Raft 是 Diego Ongaro 和 John Ousterhout 于2014年提出的共识算法，目标是与Paxos等效但更易理解。

基本思想： - Leader选举： 通过随机超时选举出一个Leader，Leader负责处理所有客户端请求和日志复制 - 日志复制： Leader接收客户端请求，将日志条目复制到所有Follower节点 - 安全性： 通过Term（任期）概念确保选举安全性 - 节点角色：Leader、Follower、Candidate

Raft相对于Paxos的优势： - 将共识问题分解为相对独立的子问题（Leader选举、日志复制、安全性） - 更强的日志匹配属性 - 更易于理解和实现

1.13.4 Paxos/Raft的容错能力

Paxos和Raft都基于多数派投票机制。

容错公式： 设总节点数为 $$n$$，最多可容忍失效节点数为 $$f$$，则：

$$n = 2f + 1$$

即：在包含 $$n$$ 个节点的分布式系统中，最多可以容忍 $$\lfloor \frac{n-1}{2} \rfloor$$ 个节点的失效。

推导：要选出多数派（过半），需要的票数至少为 $$\lfloor n/2 \rfloor + 1$$。在最坏情况下，必须有至少 $$n - f$$ 个非失效节点，且 $$n - f \geq \lfloor n/2 \rfloor + 1$$，即 $$f \leq \lfloor (n-1)/2 \rfloor$$。

示例： - 3个节点：最多容忍 1 个失效 - 5个节点：最多容忍 2 个失效 - 7个节点：最多容忍 3 个失效

注意： 以上基于失效停止模型（Fail-Stop）。如果采用拜占庭故障模型，容错公式为 $$n = 3f + 1$$（即最多容忍 $$\lfloor (n-1)/3 \rfloor$$ 个拜占庭节点）。

1.14 分布式一致性模型

分布式一致性模型定义了在分布式数据存储系统中，多个副本之间数据同步的语义保证。

1.14.1 强一致性（Strong Consistency / Linearizability）

定义： 任何读取操作都能读到最新写入的值。系统表现得像只有一个副本。

特征： - 所有操作看起来是瞬间完成的（像一个原子操作） - 写入成功后，所有后续的读取（无论访问哪个副本）都能看到该写入 - 操作之间的顺序与物理时间（墙钟）一致

实现代价： 高延迟（需要等待所有/多数副本确认），可用性受影响（部分节点失效时可能无法服务）

典型系统： Google Spanner（利用TrueTime实现）、ZooKeeper（顺序一致性近似强一致性）、etcd

1.14.2 最终一致性（Eventual Consistency）

定义： 如果系统不再有新的写入，经过足够长时间后，所有副本最终会达到一致状态。

特征： - 写入后立即读取可能读到旧数据（Stale Read） - 不保证读取到最新值 - 但保证如果停止写入，最终所有副本一致

优点： 高可用、低延迟（写入不需要等待所有副本确认）

典型系统： DNS、Amazon Dynamo、Cassandra

1.14.3 顺序一致性（Sequential Consistency）

定义： 所有操作按某种全局顺序执行（所有节点看到相同的操作顺序），且每个节点自身的操作保持程序顺序（program order）。

特征： - 比强一致性弱：不要求操作顺序与物理时间一致 - 比最终一致性强：保证全局统一的顺序 - 不需要依赖物理时钟

示例： 如果节点A先写后读，节点B先写后读，顺序一致性要求存在一个全局的操作序列，所有节点看到该序列——但不要求该序列与物理时间一致。

1.14.4 因果一致性（Causal Consistency）

定义： 具有因果关系的写入操作在所有节点上以相同的顺序被看到，但并发写入可能在不同节点上以不同顺序出现。

特征： - 比顺序一致性更弱 - 使用向量时钟跟踪因果关系 - 不保证并发操作的全局顺序

1.14.5 一致性模型强度排序（从强到弱）

强一致性（Linearizability） > 顺序一致性（Sequential） > 因果一致性（Causal） > 最终一致性（Eventual）

1.15 分布式全局唯一ID：Snowflake算法

雪花算法（Snowflake Algorithm）是Twitter开源的分布式全局唯一ID生成算法。

1.15.1 设计要求

  • 全局唯一性： 不能出现重复的ID
  • 递增性： 确保生成的ID是递增的（大致趋势递增）
  • 高可用： 在每个节点上独立运行，其他节点的失效不影响本节点的ID生成
  • 高性能： 在多服务器高并发环境下依然表现良好

1.15.2 64位ID结构

每个节点生成的ID是一个64位（bit）的 long 类型整数：

部分 位数 说明
最高位 1 bit 固定为0，保证ID为正整数
时间戳 41 bits 毫秒级时间戳，$$2^{41}/(1000 \times 60 \times 60 \times 24 \times 365) \approx 69$$ 年
机器码 10 bits 最多支持 $$2^{10} = 1024$$ 台机器
序列号 12 bits 同一毫秒内的自增序列号，每毫秒每台机器最多生成 $$2^{12} = 4096$$ 个ID

1.15.3 工作方式

  • 每个节点独立运行，无需与其他节点协调
  • 使用本机的物理时钟（毫秒级）作为时间戳
  • 同一毫秒内通过递增序列号区分不同ID
  • 如果时钟回拨，算法需要等待或报错

1.16 向量时钟应用：因果排序广播

因果排序广播（Causal Broadcast）利用向量时钟确保具有因果关系的广播消息在投递时保持因果顺序。

1.16.1 因果排序广播的性质

建立在可靠广播基础上，满足： 1. 如果消息 $$m$$ 被正确投递给节点 $$A$$，则 $$m$$ 一定会被正确投递给所有其他非失效节点 2. 对于具有因果关系的两个消息 $$m_1$$ 和 $$m_2$$（$$m_1 \rightarrow m_2$$），在任意节点上 $$m_1$$ 一定会先于 $$m_2$$ 投递给应用层

1.16.2 实现思想

中间件利用向量时钟实现因果排序广播： - 收到（Receive）消息 $$\langle m, V(m) \rangle$$ 后先放入缓存队列 - 根据向量时钟判定队列中不同消息之间的因果顺序 - 然后按因果顺序投递（Deliver）给应用层

关键区分： "Receive"（从网络收到消息入缓存）和"Deliver"（按因果顺序投递给应用层）是两个不同的动作。只有Broadcast和Deliver这才会引起向量时钟的变化，Receive不会。

二、相关真题与解析

题目1（来源：2022年）

原题（4分）： 设计分布式系统时通常采用消息传递模型，简述消息传递模型的含义。

解析/答案：

消息传递模型是分布式系统设计中最基础的抽象通信模型，其含义包括：

  • 分布式系统由多个独立的计算节点（进程/主机）组成，每个节点拥有自己独立的地址空间（独立内存）
  • 节点之间不共享内存，只能通过发送和接收消息来进行通信和协作
  • 消息通过底层通信网络在节点之间传输
  • 每个节点只知道自己本地的状态，没有全局视图，只能通过与其他节点的消息交互来推断整个系统的状态
  • 节点间的协调必须依赖消息的收发来完成

题目2（来源：2022年）

原题（2分）： 消息传递模型与设计多线程并行计算系统时采用的模型有何不同？

解析/答案：

维度 消息传递模型（分布式） 共享内存模型（多线程并行）
地址空间 各节点独立地址空间 所有线程共享同一地址空间
通信方式 通过网络收发消息 通过读写共享变量
同步方式 消息收发含隐式同步，或分布式锁 互斥锁、信号量、条件变量等
失效处理 部分节点失效是常态，必须考虑容错 通常假设不发生单线程崩溃
通信延迟 不可忽略且剧烈变化 极低且稳定
扩展性 水平扩展（增加节点）跨越物理机器 受限于单机CPU核数和内存

核心差异：消息传递模型无共享内存，通信靠网络；多线程模型共享内存，通信靠共享变量。

题目3（来源：2022年）

原题（6分）： 要设计能够真正用于实际生产环境的多计算节点分布式协作系统，必须要考虑容忍哪几类错误？

解析/答案：

必须考虑容忍以下四类错误：

  1. 不可靠的计算节点： 部分节点可能随时发生故障而失效。节点失效是常态而非异常，设计时必须假设任何节点都可能在任何时刻崩溃。

  2. 不可靠的通信网络： 通信网络可能发生丢包、数据包乱序、传输延迟剧烈变化、数据包内容被修改（被动损坏或主动攻击）、网络断裂（Partition）等问题。

  3. 不可靠的时钟： 不同节点的物理时钟因为晶振精度差异、温度、使用年限、震动等环境因素而不同步。即使初始同步，随时间推移时钟漂移会累积并超出误差范围。

  4. 没有全局观： 在分布式系统中，没有一个全局的控制中心能够实时掌握所有节点的精确状态。每个节点只能通过与其他节点的消息交互来推断系统全局状态，且这种推断总是基于延迟的、不完整的信息。

题目4（来源：2022年）

原题（4分）： 简述失效停止节点错误模型和拜占庭节点错误模型的区别？

解析/答案：

维度 失效停止模式（Fail-Stop） 拜占庭模式（Byzantine）
故障行为 节点一旦故障就停止工作，不发送任何消息（安静地停止） 节点故障后可能做出任意动作，包括发送错误/恶意消息、故意扰乱系统
故障原因 硬件故障、进程崩溃、断电等 黑客攻击、恶意代码、软件Bug、被完全控制的节点

容错公式 需 $$n > 2f$$（多数派协议如Paxos/Raft） 需 $$n > 3f$$（拜占庭容错协议）
典型应用 大多数企业级分布式系统 区块链、航天系统、安全关键系统

核心区别：失效停止的节点"不说话"；拜占庭节点的"说假话"、"说矛盾的话"，是最恶劣的故障模式。

题目5（来源：2022年）

原题（3分）： 在一个包含 $$n$$ 个节点的分布式系统中应用 Paxos 或 Raft 分布式共识协议实现不同节点间的数据同步，最多可以容忍多少个节点的失效？

解析/答案：

Paxos 和 Raft 协议都基于多数派投票（Quorum）机制。为确保系统能在节点失效时继续正常工作，需要保证剩余正常工作的节点仍能构成多数派。

设总节点数为 $$n$$，最多可容忍失效节点数为 $$f$$：

$$n - f > \frac{n}{2}$$

即 $$f < \frac{n}{2}$$，所以：

$$f_{max} = \left\lfloor \frac{n-1}{2} \right\rfloor$$

最多容忍 $$\lfloor (n-1)/2 \rfloor$$ 个节点的失效。

具体例子： - 3个节点 → 最多容忍 1 个失效（$$n = 2f + 1 = 2 \times 1 + 1$$） - 5个节点 → 最多容忍 2 个失效 - 7个节点 → 最多容忍 3 个失效

题目6（来源：2022年）

原题（6分）： 某个支持多副本模式的分布式存储系统采用了如下设计：客户端的写入操作都发送给主节点，主节点写入本地存储的同时将该请求广播给所有副本从节点。主节点等到所有副本节点返回同步成功的消息之后再给该客户端返回写入成功消息。如果某个副本节点超过规定时间没有响应，主节点就反复重试。客户端的读取请求可以发送给主节点或任意的副本从节点。请问该系统设计是否满足强一致性模型？是否满足最终一致性模型？为什么？

解析/答案：

（1）是否满足强一致性？不满足。

原因： - 强一致性（线性一致性）要求任何读取操作都能读到最新写入的值 - 该系统中，写入需要等待所有副本节点同步成功才返回 - 但读取请求可以被发送到任意的副本从节点 - 考虑以下场景：一个客户端刚完成写入（主节点和所有副本都同步成功），此时另一个客户端向某个副本从节点发起读取——由于主从同步可能存在微小的时序窗口，或网络延迟导致副本未及时接收到最新写入，该客户端的读取可能读到旧数据 - 更重要的是，强一致性要求所有操作看起来有一个全局的实时顺序，而该系统没有提供这种保证——如果读取可以选择任意副本，就无法保证永远读取到最新数据 - 另外，"反复重试"意味着系统在部分节点失效时阻塞写入，但这不等同于强一致性保证

（2）是否满足最终一致性？满足。

原因： - 最终一致性的定义是：如果系统不再有新的写入，经过足够长时间后，所有副本最终会达到一致状态 - 该系统中，主节点会将每个写入广播给所有副本，并且在副本未响应时反复重试 - 虽然可能暂时出现副本不一致（某个副本没收到最新写入），但只要系统最终恢复（副本重新上线），主节点会通过重试机制将数据同步过去 - 因此，在没有新写入的情况下，最终所有副本会达成一致状态 - 但需要注意：反复重试基于主节点的持久化日志，如果主节点在重试成功前崩溃，则需有相应的故障恢复机制

题目7（来源：2019年）

原题： 物理时钟的推导公式证明以及客户端如何调节物理时钟与服务器同步。

解析/答案：

（1）Cristian时间同步算法的推导：

设客户端C向时间服务器S发起同步请求：

  • $$T_1$$：客户端发送请求的本地时间
  • $$T_2$$：服务器收到请求的本地时间
  • $$T_3$$：服务器发送响应的本地时间（响应包携带 $$T_2$$ 和 $$T_3$$）
  • $$T_4$$：客户端收到响应的本地时间

网络往返延迟：

$$\delta_{total} = (T_4 - T_1) - (T_3 - T_2)$$

其中 $$\delta_{total} = \delta_{req} + \delta_{resp}$$（$$\delta_{req}$$ 为请求延迟，$$\delta_{resp}$$ 为响应延迟）。

客户端知道 $$\delta_{total}$$，但不知道 $$\delta_{req}$$ 和 $$\delta_{resp}$$ 各自的大小。

关键假设： 网络传输延迟在请求和响应方向大致相等，即 $$\delta_{req} \approx \delta_{resp}$$。

则有：

$$\delta_{resp} \approx \frac{\delta_{total}}{2} = \frac{(T_4 - T_1) - (T_3 - T_2)}{2}$$

服务器在 $$T_3$$ 时刻的真实时间加上响应传输延迟等于消息到达时刻的服务器时间：

$$T_{server\_at\_T_4} = T_3 + \delta_{resp} \approx T_3 + \frac{\delta_{total}}{2}$$

客户端将本地时钟调整为：

$$T_{client\_corrected} = T_3 + \frac{(T_4 - T_1) - (T_3 - T_2)}{2}$$

（2）同步误差分析（时钟偏移 Clock Skew）：

实际同步误差最大为 $$\pm \frac{\delta_{total}}{2}$$。因为：

  • 极限情况1：$$\delta_{resp} = 0$$（响应瞬间到达），则 $$\delta_{resp} = 0 < \frac{\delta}{2}$$，误差 $$\frac{\delta}{2}$$
  • 极限情况2：$$\delta_{resp} = \delta$$（请求瞬间到达，响应占全部延迟），误差也为 $$\frac{\delta}{2}$$

$$|T_{client\_corrected} - T_{server\_actual}| \leq \frac{\delta_{total}}{2}$$

（3）客户端调整时钟的策略（NTP采用的方法）：

  • 策略一：微调本地时钟频率（Slewing）——稍微加快或减慢时钟频率，使本地时钟逐渐校准。这是最常用的方法，不会引起时间跳跃。
  • 策略二：直接设置时钟值（Stepping）——当时钟偏差过大时直接跳变。缺点是会导致时间不连续。
  • 策略三：报错——当时钟偏差过大无法自动修复时通知管理员。

题目8（来源：2019年）

原题： 给出三个节点，一系列消息发送/接收之后，最终各节点的逻辑时钟值。并回答：该排序的性质：保持了潜在因果关系：a导致b，则一定有C(a)<C(b)，但反过来不成立——为什么？

解析/答案：

（1）Lamport逻辑时钟计算示例：

设有三个节点 $$P_1, P_2, P_3$$，初始逻辑时钟均为 0。

事件序列如下（假设按时间顺序发生）：

  1. $$P_1$$ 发生内部事件 a：$$C_1(a) = 1$$
  2. $$P_1$$ 向 $$P_2$$ 发送消息 m1（时间戳 = 1）：$$C_1 = 2$$（发送也是事件，自增）
  3. $$P_2$$ 接收 m1（时间戳为 2）：$$C_2 = 1 + \max(0, 2) = 3$$
  4. $$P_2$$ 发生内部事件 b：$$C_2(b) = 4$$
  5. $$P_3$$ 发生内部事件 c：$$C_3(c) = 1$$
  6. $$P_2$$ 向 $$P_3$$ 发送消息 m2（时间戳 = 4）：$$C_2 = 5$$
  7. $$P_3$$ 接收 m2（时间戳为 5）：$$C_3 = 1 + \max(1, 5) = 6$$
  8. $$P_1$$ 发生内部事件 d：$$C_1(d) = 3$$

最终各节点逻辑时钟： - $$P_1: 3$$ - $$P_2: 5$$ - $$P_3: 6$$

（2）为什么"反过来不成立"？

Lamport逻辑时钟定理： - 正向： 如果 $$a \rightarrow b$$（a先于b，有因果关系），则一定有 $$C(a) < C(b)$$ - 反向： 如果 $$C(a) < C(b)$$，则不一定有 $$a \rightarrow b$$

反例：在上面的例子中，$$C_3(c) = 1$$，$$C_1(a) = 1$$，$$C_1(d) = 3$$。

  • $$C_1(a) < C_3(c)$$ 吗？$$1 < 1$$ 为假；$$C_1(a) = C_3(c) = 1$$，不满足严格小于。
  • $$C_1(d) = 3 > C_3(c) = 1$$，所以 $$C_3(c) < C_1(d)$$，但事件 c（在 $$P_3$$ 上）和事件 d（在 $$P_1$$ 上）之间没有任何因果关系——它们是并发事件（$$c \parallel d$$）。

更清晰的例子： 假设 $$P_1$$ 发生了内部事件 e（$$C_1(e)=5$$），同时 $$P_2$$ 发生了内部事件 f（$$C_2(f)=3$$）。显然 $$C_2(f) < C_1(e)$$，但是 f 和 e 是完全独立的事件，没有任何因果关系（没有消息在它们之间传递）。它们只是碰巧在不同的时钟速率下获得了不同大小的时钟值。

根本原因： Lamport标量逻辑时钟只有一个整数维度，将因果关系的丰富信息"压缩"到了一个标量中，丢失了因果关系图的结构信息。因此只能保证"因果关系→时钟小"的单向蕴含，无法反向推断。

题目9（逻辑时钟变式练习：全序时钟排序）

参考2019年考试中提到"保持了潜在因果关系：a导致b，则一定有C(a)<C(b)，但反过来不成立"。该性质对全序逻辑时钟同样成立。下面给出全序逻辑时钟的排序示例。

题： 系统中有3个节点（ID分别为 1, 2, 3），已知以下事件的全序时间戳： - 事件 a：$$\langle 2, 1 \rangle$$（发生在节点1） - 事件 b：$$\langle 2, 2 \rangle$$（发生在节点2） - 事件 c：$$\langle 3, 1 \rangle$$（发生在节点1）

请按全序逻辑时钟规则对这三个事件排序。

解析：

全序排序规则：$$\langle C(a), ID(a) \rangle < \langle C(b), ID(b) \rangle \iff C(a) < C(b) \text{ 或 } (C(a) = C(b) \text{ 且 } ID(a) < ID(b))$$

  • 比较 a 和 b：$$C(a) = 2 = C(b)$$，触犯相等判断，比较 ID：$$ID(a)=1 < ID(b)=2$$，所以 $$a < b$$
  • 比较 b 和 c：$$C(b) = 2 < C(c) = 3$$，所以 $$b < c$$
  • 比较 a 和 c：$$C(a) = 2 < C(c) = 3$$，所以 $$a < c$$

全序结果： $$a < b < c$$

注意： 即使 $$a < b$$ 按全序排在前面，a 和 b 之间也可能没有因果关系（因为它们发生在不同节点且时钟值相同，说明它们很可能是并发事件）。这正是"反过来不成立"的含义。

题目10（来源：2022年）

原题（4分）： 要利用分布式计算系统对一个大规模数据集进行排序，你所知道的排序算法中哪一种更适合？为什么？

解析/答案：

归并排序（Merge Sort）更适合分布式环境下的排序。

原因： 1. 天然的分治结构： 归并排序的"分而治之"策略与分布式系统的架构天然匹配。可以先将数据分布到各节点分别排序，再进行多路归并。 2. 外部排序兼容： 大规模数据集可能无法全部放入单机内存，归并排序天然支持外部排序（external sort）。 3. 可并行化： 每个节点可以独立地对分配给自己的数据分区进行排序（Map阶段），然后通过多路归并（Reduce阶段）产生全局有序结果。 4. 通信模式规整： 在归并阶段，通信模式是可预测的（如从各节点拉取有序分区的数据），可以有效利用网络带宽。 5. 实际系统验证： Google的MapReduce、Apache Hadoop、Apache Spark中实现大规模排序时，都采用基于归并排序思想的算法（如TeraSort）。

不适合的算法示例： - 快速排序（Quick Sort）：在分布式环境中，递归的partition操作需要全局数据视图，通信开销大 - 冒泡/插入等 $$O(n^2)$$ 算法：数据规模大时效率极低

  1. 消息传递模型是基础概念，需要理解其与共享内存模型的本质区别
  2. Happen-Before关系的定义和三条规则必须熟记
  3. Lamport逻辑时钟算法是重点，包括：事件处理规则、发送/接收时的时钟更新公式、性质（因果保持但反向不成立）

  6. 全序逻辑时钟：理解二元组$$\langle C, ID \rangle$$的排序规则
  7. 广播协议分类要清楚：可靠广播→FIFO广播→因果广播→全序广播，从弱到强的关系
  8. Paxos/Raft容错：$$n$$个节点最多容忍$$\lfloor (n-1)/2 \rfloor$$个失效（基于多数派）
  9. 一致性模型：能区分强一致性、最终一致性、顺序一致性，并分析给定场景是否满足

分布式系统 · 知识点精讲版

一、分布式系统核心概念

1.1 定义与五要素

多个网络互联、独立自治的计算节点，基于消息传递协作完成共同目标。

五要素关键点
多个计算节点进程/线程/虚拟机均可
网络互联不限物理拓扑
独立自治独立CPU、独立时钟、独立失效
消息传递非共享内存（Share Nothing）
共同目标单节点无法独立完成

必记辨析：

  • 多CPU单主板计算机 不是 分布式系统（共享内存 SMP，非消息传递）
  • 同一物理机 可以 包含多个分布式节点（进程/虚拟机 + loopback 通信）

1.2 共享内存 vs 消息传递

共享内存消息传递
通信隐式（读写共享变量）显式（send/recv）
耦合紧耦合松耦合
容错差（单点崩溃全局影响）好（独立失效）
扩展受限于总线带宽理论无限水平扩展

1.3 分布式系统必须容忍的四类错误（真题高频）

  1. 不可靠的计算节点：任何节点可能随时崩溃失效，失效是常态
  2. 不可靠的通信网络：丢包、乱序、延迟抖动、数据损坏、网络分区
  3. 不可靠的时钟：不同节点物理时钟因晶振差异、温度等逐渐漂移
  4. 没有全局观：不存在全局控制中心实时掌握所有节点状态，只能通过消息交互推断

1.4 设计目标

  • 透明性：用户感知不到分布（位置透明、访问透明、故障透明等）
  • 可扩展性：水平扩展（加节点）提升性能/容量
  • 容错性：部分节点/网络故障时系统仍能提供服务
  • 一致性：多副本数据在可接受范围内保持一致

三、通信技术

3.1 RPC 远程过程调用

核心思想：让远程函数调用像本地调用一样简单。

完整流程：

  1. Client 调用本地 Client Stub
  2. Client Stub 将参数序列化，通过网络发送给 Server
  3. Server 端的 Server Skeleton 接收并反序列化参数
  4. Skeleton 调用实际 Server 函数
  5. Server 函数返回结果给 Skeleton
  6. Skeleton 序列化结果，通过网络返回给 Client Stub
  7. Client Stub 反序列化结果，返回给 Client
  • Stub：客户端代理，负责序列化参数、发起网络调用
  • Skeleton：服务端代理，负责反序列化参数、调用实际函数
  • IDL：接口定义语言，支持跨语言调用

RPC 中间件作用：屏蔽网络细节、序列化/反序列化、跨语言、服务注册发现。

3.2 消息中间件（MOM）

模式特点
消息队列 (Queue)点对点，一个消息一个消费者，FIFO
发布/订阅 (Topic)一对多，一个消息多个订阅者

消费者接收消息三种方式：同步接收（阻塞等待）、异步接收（回调）、轮询接收。

MOM 优点：解耦、削峰填谷、异步、可靠投递。

秒杀/双十一：请求入队列 → 后端按能力消费 → 保护后端不被冲垮。

3.3 并发服务技术

技术适用场景
多线程通用，连接数有限
线程池复用线程
事件驱动 (Reactor)大量短连接
Proactor异步I/O
协程高并发、轻量级

4.3 Happen-Before 关系（先于关系）

记作 $a \rightarrow b$，是分布式事件间的偏序关系，由 Lamport 提出。

四条规则：

  1. 同一进程内：若事件 $a$ 和 $b$ 在同一节点，且 $a$ 在 $b$ 之前发生，则 $a \rightarrow b$
  2. 消息传递：若 $a$ 是"$P_1$ 发送消息 $m$ 到 $P_2$"，$b$ 是"$P_2$ 收到该消息"，则 $a \rightarrow b$
  3. 传递性：若 $a \rightarrow b$ 且 $b \rightarrow c$，则 $a \rightarrow c$
  4. 并发事件：若 $a \nrightarrow b$ 且 $b \nrightarrow a$，则 $a$ 和 $b$ 并发，记作 $a \parallel b$

关键性质：$\rightarrow$ 隐含因果关系。若 $a \rightarrow b$，则 $a$ 可能因果影响了 $b$。

4.4 Lamport 逻辑时钟（标量时钟）

为什么需要逻辑时钟：物理时钟很难精确同步，分布式系统只需对事件顺序达成一致，且尽量保持因果关系即可。

算法规则（设 $C_i$ 为节点 $i$ 的本地逻辑时钟）：

  1. 内部事件/发送事件：节点 $i$ 中每发生一个新事件，$C_i \leftarrow C_i + 1$
  2. 发送消息：节点 $i$ 发出的每个消息都绑定当前的 $C_i$ 作为时间戳
  3. 接收消息：节点 $j$ 收到时间戳为 $C_{msg}$ 的消息后：$C_j \leftarrow 1 + \max(C_j, C_{msg})$

解释：接收方将自己的时钟设为"自身时钟和消息时间戳的较大值加1"，保证接收事件时间戳一定大于发送事件时间戳。

三条性质：

  1. 保持因果关系：若 $a \rightarrow b$，则一定有 $C(a) < C(b)$
  2. 反向不成立：$C(a) < C(b)$ 时，不一定有 $a \rightarrow b$（可能是并发事件碰巧时钟小）
  3. 可能相等：两个不同并发事件可能获得相同时间戳

4.5 全序逻辑时钟

动机：Lamport 标量时钟在 $C(a)=C(b)$ 时无法区分先后顺序，全序时钟将时间戳扩展为二元组，实现所有事件全序排列。

时间戳格式：每个事件携带二元组 $\langle C, ID \rangle$，其中 $C$ 是标量逻辑时钟值，$ID$ 是节点编号。

排序规则：

$$L(a) < L(b) \iff C(a) < C(b) \;\mathrm{OR}\; \Big(C(a)=C(b) \;\mathrm{AND}\; ID(a)<ID(b)\Big)$$

算法描述：

  • 每个节点维护标量时钟 $C$
  • 节点内每发生新事件，$C \leftarrow C + 1$
  • 发送消息时携带 $\langle C, ID \rangle$
  • 接收到时间戳 $\langle C_i, i \rangle$ 后：$C_j \leftarrow 1 + \max(C_j, C_i)$

性质：保持潜在因果关系（$a \rightarrow b$ 则 $L(a) < L(b)$），但反向仍不成立。

4.6 向量逻辑时钟

基本思想：设系统有 $n$ 个节点，节点 $N_i$ 维护一个 $n$ 维向量：

$$\mathbf{T}_i = \langle t_1, t_2, \ldots, t_n \rangle$$

  • $\mathbf{T}_i[i]$：节点 $N_i$ 自己的标量逻辑时钟
  • $\mathbf{T}_i[j]\;(j \neq i)$：节点 $N_i$ 所知道的关于节点 $N_j$ 的最大标量时钟值

直观理解：每个节点维护了一个"我认为每个节点现在进展到哪一步了"的全局视图。

算法四步：

  1. 初始化：所有节点向量时钟初始化为 $\langle 0, 0, \ldots, 0 \rangle$
  2. 内部事件：节点 $N_i$ 内每发生一个新事件，$\mathbf{T}_i[i] \leftarrow \mathbf{T}_i[i] + 1$
  3. 发送消息：节点 $N_i$ 发送消息时，将当前向量时钟 $\mathbf{T}_i$ 作为时间戳附加
  4. 接收消息：收到来自节点 $j$ 的消息 $\langle m, \mathbf{T}_m \rangle$ 后：
    融合：对所有 $k = 1, \ldots, n$，$\mathbf{T}_i[k] \leftarrow \max(\mathbf{T}_i[k], \mathbf{T}_m[k])$
    自增：$\mathbf{T}_i[i] \leftarrow \mathbf{T}_i[i] + 1$

向量时钟的比较与排序：

  • 相等：$\mathbf{T} = \mathbf{T}' \iff \forall j: \mathbf{T}[j] = \mathbf{T}'[j]$
  • 小于等于：$\mathbf{T} \leq \mathbf{T}' \iff \forall j: \mathbf{T}[j] \leq \mathbf{T}'[j]$
  • 严格小于：$\mathbf{T} < \mathbf{T}' \iff \mathbf{T} \leq \mathbf{T}' \text{ 且 } \mathbf{T} \neq \mathbf{T}'$
  • 并发（不可比较）：$\mathbf{T} \parallel \mathbf{T}' \iff \mathbf{T} \nleq \mathbf{T}' \text{ 且 } \mathbf{T}' \nleq \mathbf{T}$

关键性质：

  • $a \rightarrow b \iff V(a) < V(b)$（充要条件！这是向量时钟的核心优势）
  • $a \parallel b \iff V(a) \parallel V(b)$

4.8 Cristian 时间同步算法

背景：分布式系统需要同步物理时钟，但网络延迟不可忽略。Cristian 算法通过向时间服务器请求来估计并校正本地时钟。

四个时间戳：

  • $T_1$：客户端发送请求的本地时间
  • $T_2$：服务器收到请求的本地时间
  • $T_3$：服务器发送响应的本地时间（响应携带 $T_2, T_3$）
  • $T_4$：客户端收到响应的本地时间

网络往返延迟：

$$\delta_{total} = (T_4 - T_1) - (T_3 - T_2)$$

其中 $\delta_{total} = \delta_{req} + \delta_{resp}$。

关键假设：网络传输延迟在请求和响应方向大致相等，$\delta_{req} \approx \delta_{resp} \approx \delta_{total}/2$。

客户端校时公式：

$$T_{client} \leftarrow T_3 + \frac{(T_4 - T_1) - (T_3 - T_2)}{2}$$

误差分析：实际误差最大为 $\pm \delta_{total}/2$。

NTP 采用的三种调整策略：

  1. Slewing（微调频率）：稍微加快或减慢时钟频率，逐渐校准，不引起时间跳跃，最常用
  2. Stepping（直接设置）：偏差过大时直接跳变，会导致时间不连续
  3. 报错：偏差过大无法自动修复时通知管理员

4.9 一致性模型

模型定义典型场景
强一致性（线性一致性）任何读取都能读到最新写入的值，所有操作看起来有一个全局实时顺序银行转账、库存扣减

五、MapReduce 与大数据生态

5.1 HDFS 分布式文件系统

HDFS（Hadoop Distributed File System）是面向大数据的开源分布式文件系统，将多个节点的存储资源整合，向用户呈现统一的目录树。

核心特性：

  • 大文件自动分块（Block），默认每块 128MB
  • 每个 Block 默认存 3 个副本，分布在不同节点
  • 适合一次写入多次读取，不适合大量小文件

5.1.1 NameNode 与 DataNode

角色功能
NameNode（主节点）存储元数据（目录、文件名、Block 列表、Block→DataNode 映射）；管理 DataNode 注册与心跳；负责多副本维护和负载均衡；处理控制流请求
DataNode（从节点）存储实体 Block 数据；处理数据流读写；定期向 NameNode 发送心跳和 Block 报告；执行复制/删除指令

控制流与数据流分离：

  • 控制流：客户端 → NameNode（元数据操作）
  • 数据流：客户端 → DataNode（实际数据读写）

5.1.2 客户端读取 HDFS 文件的流程

  1. 客户端向 NameNode 发送读取请求，传入文件路径
  2. NameNode 返回该文件的 Block 列表及每个 Block 所在 DataNode 地址（按距客户端由近到远排序）
  3. 客户端选择最近的 DataNode，读取第一个 Block
  4. 依次读取后续 Block，直到文件读完
  5. 若某个 DataNode 失败，从备选列表选另一个 DataNode 重读
  6. 客户端将所有 Block 组装成完整文件

5.1.3 数据可靠性机制

  • 多副本机制：默认 3 副本，副本不足时 NameNode 自动触发复制
  • 机架感知：副本放置策略——第1个副本放本地节点，第2个放同一机架另一节点，第3个放不同机架。兼顾写带宽和容错
  • 心跳机制：DataNode 每 3 秒向 NameNode 发心跳，超过 10 分钟未收到则标记失效并补副本
  • 安全模式：NameNode 启动后等待 DataNode 上报 Block 信息，确认完整后退出
  • 校验和：写入时计算 Block 校验和，读取时验证
  • 元数据持久化：FsImage（命名空间快照）+ EditLog（操作日志）

5.1.4 小文件问题

为什么 HDFS 不适合大量小文件？

  1. NameNode 内存压力：每个文件、目录、Block 的元数据（约 150 字节/条目）都存 NameNode 内存，大量小文件会耗尽内存
  2. MapReduce 效率低：每个文件分片对应一个 Map Task，小文件产生大量 Map Task，调度开销远大于数据处理本身
  3. 寻道时间占比大：每个 Block 读取都要磁盘寻道

解决方案：SequenceFile、HAR（Hadoop Archive）、Parquet/ORC 等合并小文件。

5.2 MapReduce 工作流程

核心思想：将大数据计算任务划分为 Map、Shuffle、Reduce 三个阶段。

五阶段详细流程：

  1. 输入分片（Split）：将输入文件划分为多个 Split，每个 Split 交给一个 Map Task
  2. Map 阶段：从 Split 解析 K-V 对，对每个 K-V 调用一次 Map 方法，输出中间 K-V 对
  3. 二次分区与排序：对 Map 输出按 Key 二次分区，每个分区内按 Key 排序
  4. Shuffle/Combine：Reduce Task 从各 Map 节点拉取属于自己分区的数据，合并排序
  5. Reduce 阶段：对每个 Key 调用一次 Reduce 方法，输出最终 K-V 对

Shuffle 的本质：按 Key 二次分区，把相同 Key 的中间结果聚集到同一个 Reduce 节点。不是随机打乱。

键值数组转换关系：

  • 输入键值数组 X 经 Map 转为中间键值数组 Y
  • Y 经 Reduce 转为最终键值数组 Z
  • X、Y、Z 的 Key/Value 类型可以不同

5.3 Spark 计算模型

对比维度MapReduceSpark
计算模型基于磁盘的批处理基于内存的 DAG 计算
中间结果写入 HDFS 磁盘优先存内存
速度较慢快（MR 的 10-100 倍）
适用场景一次性大规模批处理迭代计算、交互式查询、流处理
编程模型仅 Map、Reduce丰富的 Transformation 和 Action

RDD（弹性分布式数据集）：Spark 的基本抽象，是只读的、可分区的分布式数据集合。

  • Transformation（转换）：如 map、filter、reduceByKey，懒加载，不会立即执行
  • Action（动作）：如 collect、count、saveAsTextFile，触发真正的 DAG 执行

5.4 一致性哈希

基本思想：把服务器和数据都哈希到一个环上，每条数据顺时针遇到的第一个服务器就是存储节点。

关键计算步骤：

  1. 计算各服务器哈希值，在环上排序
  2. 确定每台服务器负责的区间：从上一台服务器下一位置到自身
  3. 计算各数据哈希值，按"顺时针第一个服务器"规则分配

增减节点：只影响相邻区间，只移动相邻区间的数据，不会全部重新分布。

虚拟节点：为每台服务器分配多个虚拟节点，解决数据倾斜问题。

六、广播协议与共识

6.1 广播协议

可靠广播（Reliable Broadcast）

  1. 有效性：若正确节点广播消息 $m$，则其他所有正确节点最终都能收到 $m$
  2. 一致性/统一性：若 $m$ 被投递给某个正确节点，则 $m$ 一定被投递给所有其他非失效节点
  3. 完整性：正确节点收到的 $m$ 一定是某个节点发送的，不能"无中生有"

全序广播（Total Order Reliable Broadcast）

在可靠广播基础上，所有节点以完全相同的顺序投递多个广播消息。

FIFO 广播（FIFO Reliable Broadcast）

在可靠广播基础上，若同一个节点先广播 $m_1$ 后广播 $m_2$，则其他节点也先收到 $m_1$ 后收到 $m_2$。

6.2 Paxos 共识协议

三角色：

  • Proposer（提议者）：提出提案
  • Acceptor（接受者）：对提案投票
  • Learner（学习者）：学习已通过的决议

两阶段：

  1. Prepare/Promise 阶段：Proposer 发送 Prepare(n) 给 Acceptor；Acceptor 承诺不再接受编号小于 $n$ 的提案，返回已接受的编号最大提案
  2. Accept/Accepted 阶段：Proposer 根据 Promise 回复构造提案值，发送 Accept(n, v)；Acceptor 接受该提案并广播 Accepted

容错公式：

$$f_{max} = \left\lfloor \frac{n-1}{2} \right\rfloor$$

即 $n$ 个节点最多容忍 $\lfloor (n-1)/2 \rfloor$ 个失效，需要多数派（$n - f > n/2$）存活。

6.3 Raft 共识协议

三角色：

  • Leader：处理所有客户端请求，负责日志复制
  • Follower：接收并持久化 Leader 的日志
  • Candidate：Follower 超时未收到 Leader 心跳时发起选举

选举流程：

  1. Follower 超时未收到心跳，变为 Candidate，自增任期号，投票给自己
  2. Candidate 向其他节点请求投票
  3. 节点收到投票请求后，若该任期未投过票且 Candidate 日志足够新，则投票
  4. 获得多数派投票后成为 Leader

日志复制：

  1. Leader 接收客户端请求，追加到本地日志
  2. Leader 发送 AppendEntries 给 Follower
  3. Follower 确认并追加日志
  4. 超过半数 Follower 确认后，Leader 提交该日志并响应客户端

Raft 与 Paxos 的关系：Raft 等价于 Paxos，但设计更易理解和实现。

七、一致性模型（从强到弱）

模型强度含义典型场景
强一致性（线性一致性）最强写后立即可读最新，全局顺序银行转账、库存扣减
顺序一致性强按程序顺序，不同节点顺序可不同但各自一致分布式数据库会话
因果一致性中等有因果关系的事件顺序一致社交网络评论
最终一致性弱无新写入时最终一致DNS、商品评论、缓存

判断技巧：若系统写入需等待所有副本同步后才返回，且读取只能读主节点 → 可能强一致；若读取可任意读副本 → 通常是最终一致。

十、典型例题（带完整解答）

例题1：Lamport 逻辑时钟计算

题目：有三个节点 $P_1, P_2, P_3$，初始逻辑时钟均为 0。事件序列如下：

  1. $P_1$ 发生内部事件 a
  2. $P_1$ 向 $P_2$ 发送消息 m1
  3. $P_2$ 接收 m1
  4. $P_2$ 发生内部事件 b
  5. $P_3$ 发生内部事件 c
  6. $P_2$ 向 $P_3$ 发送消息 m2
  7. $P_3$ 接收 m2
  8. $P_1$ 发生内部事件 d

求：每个事件的 Lamport 时间戳，并说明"反向不成立"的含义。

解答：

  • (1) $P_1$ 内部事件 a：$C_1$ 从 0 自增 → $C_1(a) = 1$
  • (2) $P_1$ 发送 m1：发送也是事件 → $C_1$ 再自增，m1 时间戳 = 2
  • (3) $P_2$ 接收 m1：$C_2 = 1 + \max(0, 2) = 3$
  • (4) $P_2$ 内部事件 b：$C_2(b) = 4$
  • (5) $P_3$ 内部事件 c：$C_3(c) = 1$
  • (6) $P_2$ 发送 m2：m2 时间戳 = 5
  • (7) $P_3$ 接收 m2：$C_3 = 1 + \max(1, 5) = 6$
  • (8) $P_1$ 内部事件 d：$C_1(d) = 3$

最终结果：$C_1(a)=1, C_1(d)=3$；$C_2$ 最终 = 5；$C_3(c)=1, C_3$ 最终 = 6。

反向不成立的反例：$C_3(c) = 1 < C_1(d) = 3$，但事件 c（在 $P_3$ 上）和事件 d（在 $P_1$ 上）之间没有任何消息传递，是并发事件。因此 $C(a) < C(b)$ 不能推出 $a \rightarrow b$。

例题2：全序逻辑时钟排序

题目：系统中有 3 个节点（ID 分别为 1, 2, 3），已知事件的全序时间戳：

  • 事件 a：$\langle 2, 1 \rangle$（发生在节点 1）
  • 事件 b：$\langle 2, 2 \rangle$（发生在节点 2）
  • 事件 c：$\langle 3, 1 \rangle$（发生在节点 1）

求：按全序逻辑时钟规则排序。

解答：

  • 比较 a 和 b：$C(a)=C(b)=2$，比较 ID：$1 < 2$，所以 $a < b$
  • 比较 b 和 c：$C(b)=2 < C(c)=3$，所以 $b < c$
  • 比较 a 和 c：$C(a)=2 < C(c)=3$，所以 $a < c$

全序结果：$a < b < c$。

注意：全序只保证所有事件有唯一顺序，不代表 $a < b$ 就一定有因果关系。若 $C(a)=C(b)$，说明它们很可能是并发的。

例题3：一致性哈希分配（N=3）

题目：设 $N=3$，$H(ID_1)=4, H(ID_2)=23, H(ID_3)=13$。10 条记录的哈希值为 $1, 3, 5, 9, 14, 15, 19, 21, 26, 31$，哈希空间为 $[0, 31]$。求每台服务器分别存储几条记录。

解答：

第一步：服务器按哈希值排序：

  • $ID_1$：位置 4
  • $ID_3$：位置 13
  • $ID_2$：位置 23

第二步：确定负责区间（顺时针方向，从上一节点下一位置到自身）：

  • $ID_1$（4）：区间 $(23, 31] \cup [0, 4]$
  • $ID_3$（13）：区间 $(4, 13]$
  • $ID_2$（23）：区间 $(13, 23]$

第三步：分配记录：

  • $ID_1$：1, 3, 26, 31 → 4 条
  • $ID_3$：5, 9 → 2 条
  • $ID_2$：14, 15, 19, 21 → 4 条
例题4：一致性哈希增加节点后的迁移量

题目：接例题3，增加服务器 $ID_4$，$H(ID_4)=30$。有多少条记录需要迁移？

解答：

新增 $ID_4$（位置 30），顺时针方向下一台服务器是 $ID_1$（位置 4，环状），因此 $ID_4$ 接管 $ID_1$ 的区间 $(23, 30]$。

检查原属 $ID_1$ 的记录：1, 3, 26, 31。

  • 1, 3：不在 $(23, 30]$，仍属 $ID_1$
  • 26：在 $(23, 30]$ 内，迁移到 $ID_4$
  • 31：大于 30，仍顺时针属 $ID_1$

迁移量：1 条（记录 26）。

关键结论：一致性哈希增节点只移动相邻区间的数据，不会导致全局重分布。

11.1 MapReduce 万能骨架

适用题目：WordCount、粉丝统计、工资统计、TopN、倒排索引、grandchild-grandparent 等 12 分大题。

class XxxMapper:
    method map(LongWritable key, Text value):
        // 1. 拆行
        tokens = value.toString().split(",")
        // 2. 按题目组装 key 和 value
        outKey = 组装成你想聚合的键       // 单字段或 "A\tB" 复合键
        outVal = 组装成你要累加/统计的值   // 通常是 1 或 某个数值
        context.write(new Text(outKey), new IntWritable(outVal))

class XxxReducer:
    method reduce(Text key, Iterable<IntWritable> values):
        // 3. 聚合
        sum = 0; count = 0
        for each v in values:
            sum += v.get();  count++
        // 4. 输出
        result = 按题目要求计算(sum, count)   // 求和/平均/列表拼接
        context.write(key, new Text(result))

// 可选 Combiner：若只是求和，可直接复用 Reducer
job.setCombinerClass(XxxReducer.class)

套用要点：

  • Map 只做一件事：把每行解析成 (聚合键, 1 或 数值)
  • Reduce 只做一件事：对相同 key 的 values 累加/拼接
  • 需要两轮 MapReduce 时（如 grandchild-grandparent），第一轮的输出作为第二轮的输入
  • 求平均：Reduce 里同时记 sum 和 count，最后除

11.2 Spark 通用骨架

适用题目：所有 Spark 12 分大题（图度数、成绩分段、数据过滤统计等）。

// 1. 创建 SparkContext
SparkConf conf = new SparkConf().setAppName("AppName")
JavaSparkContext sc = new JavaSparkContext(conf)

// 2. 读取数据创建 RDD
JavaRDD<String> lines = sc.textFile("hdfs://input/path")

// 3. Transformation 链（惰性，不立即执行）
result = lines
    .filter(line -> 过滤条件)              // 过滤掉不需要的行
    .map(line -> 解析出某个字段)            // 映射
    .mapToPair(x -> new Tuple2(key, 1))    // 转成 (key, 1) 键值对
    .reduceByKey((a, b) -> a + b)          // 按 key 聚合求和
    .filter(tuple -> tuple._2 > 阈值)    // 二次过滤

// 4. Action 触发计算
result.saveAsTextFile("hdfs://output/path")
// 或 result.collect() / result.count()

类型算子作用
Transformationmap一对一映射
filter过滤
flatMap一对多展开

reduceByKey按 key 聚合（推荐，有预聚合）
groupByKey按 key 分组（效率低，慎用）

11.3 Spark 双统计合并模板（出度+入度类）

适用题目：图节点度数、双向关系统计、需要同时统计两个视角再合并的场景。

方案一：flatMap 展开法（推荐，最简洁）——每条记录展开成两条，再 reduceByKey。

// 每条边同时贡献 出度(给src) 和 入度(给dst)
JavaPairRDD<String, Integer> allDegrees = edges
    .flatMapToPair(edge -> {
        return Arrays.asList(
            new Tuple2<String, Integer>(edge.src, 1),  // 出度
            new Tuple2<String, Integer>(edge.dst, 1)   // 入度
        )
    })
    .reduceByKey((a, b) -> a + b)
    .filter(tuple -> tuple._2 > 1000)   // 过滤度数 > 1000
allDegrees.saveAsTextFile("hdfs://output/path")

方案二：分两次计算 + union 合并

outDeg = edges.mapToPair(e -> (e.src, 1)).reduceByKey((a,b) -> a+b)
inDeg  = edges.mapToPair(e -> (e.dst, 1)).reduceByKey((a,b) -> a+b)
total  = outDeg.union(inDeg).reduceByKey((a,b) -> a+b)
                  .filter(t -> t._2 > 1000)

套用要点：

  • 有向边 <ID1, ID2, Weight>：ID1 得出度，ID2 得入度
  • Total = outDeg + inDeg，每条边同时贡献一个出度和一个入度
  • 优先用 flatMap 展开法，一轮搞定

真题完整解法见：04 题目12 图节点度数（12分）

11.4 Spark 分段统计模板（分类计数）

适用题目：成绩分段、年龄段统计、按区间分桶计数等。

// 1. 先算出每个对象的指标值（如平均分）
JavaRDD<Double> scores = lines
    .filter(line -> 只保留需要的行)
    .mapToPair(line -> (姓名, 成绩))
    .groupByKey()                      // 或 reduceByKey / aggregateByKey
    .map(tuple -> sum/count)            // 算平均

// 2. 分桶：把指标映射成 (区间标签, 1)
JavaPairRDD<String, Integer> distribution = scores
    .mapToPair(avg -> {
        if (avg >= 90) return new Tuple2("90~100", 1)
        else if (avg >= 80) return new Tuple2("80~89", 1)
        else if (avg >= 70) return new Tuple2("70~79", 1)
        else if (avg >= 60) return new Tuple2("60~69", 1)
        else return new Tuple2("60以下", 1)
    })
    .reduceByKey((a, b) -> a + b)       // 各区间人数

// 3. 输出
distribution.saveAsTextFile("hdfs://output/path")

套用要点：

  • 分桶套路固定：mapToPair 用 if/else 把数值映射成 (标签, 1)，再 reduceByKey 计数
  • 若要先按学生聚合各科成绩再算平均，用 groupByKey 或 aggregateByKey 求 (sum, count)
  • 区间标签作为 key，这样 reduceByKey 才能把同区间的人加起来

真题完整解法见：04 题目11 学生成绩分段（12分）

十二、易错点速记

  1. Lamport 逻辑时钟：$C(a) < C(b) \nRightarrow a \rightarrow b$（反向不成立！）
  2. 向量时钟：$V(a) < V(b) \iff a \rightarrow b$（充要条件）
  3. 全序时钟：$ID$ 仅用于打破 $C$ 相等时的平局，不代表因果关系
  4. Fail-Stop vs Byzantine：容错公式不同（$n > 2f$ vs $n > 3f$）
  5. Cristian 算法：误差最大为 $\pm \delta/2$
  6. 一致性哈希：增节点只移动相邻区间数据，非全部重新分布
  7. HDFS 小文件：瓶颈在 NameNode 内存，非 DataNode 磁盘
  8. MapReduce Shuffle：按 Key 二次分区，非随机打乱
  9. RPC vs MOM：RPC 同步请求-响应；MOM 异步消息
  10. MPI 阻塞：Send 完成后缓冲区可复用；非阻塞需 Wait 确认
